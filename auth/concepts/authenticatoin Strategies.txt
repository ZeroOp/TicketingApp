There is no perfect solutions for this problem in microservices app. 

Option 1:
[
 req : {
    body: {},
    token : {JWT, Cookie, ETC}
 }
]

when the requests hits to the current service we can rely or depend on the auth service and make a sync call. 

Auth servicer logic to inspect JWT/Cookie and decide if user is authenticated or not. 

Here direct request to another service. 

The down side has the similar to the synchrous approach down sides . 

What happens if the authentication service is down. 

Then whole system will go down. No service don't know weather a person is logged in or not. 

Auth request will allways fail when auth service goes down. 

Next option 1.1. 

between the request and needed service like order service keep the auth service in between. 

Request => AuthService => OrderService. 

Here also the problem is same. Because we are relying on the auth service. 



Option 2: 

We are goint to teach each service weather the user is authenticated or not. 

Req => OrderSerice[Logic to inspect JWT/Cookie and decide if user is authenticated ? ]

Here no dependency on the outside sources. Everything is wrapped inside one single service. 

Upside:
   no dependency
   faster
Downside:
   * We end up duplicating the same logic across all the services . (checking logic).(That is not a big deal we can move this authentication logic into another
   shared library that used among all the services. ) (duplication is not problem).
   

Huge down sides to Option 2:

User ABC =>Req [body: {email, password}] => AuthService[ Email good? Password good? , yes? great. Here's a token to idendify you. Include it on all followup requests ].

Auth service => JWT, Cookie , whatever => UserABC. 


UserABC => OrderService[ ] since the user is logged in they can purchase a ticket. No communication service. 


What might happened after userABC got the token. 

AdminUser => Request to BAN a User['id' : 'UserABC'] => Auth service [ User Management Logic [ Ban UserABC ? Ok , i will update the database ] ] => Auth MongoDB[ {id: 'UserABC' : hasAccess : true}]

So if the userABC makes another follow up request to the Orderservice, we want to revoke the request but order service don't know. 

UserABC is authenticated. 

That the core issue with this approach.
